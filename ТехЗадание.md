ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ)Разработка автоматизированной системы учета, оценки и мониторинга рисков «RiskTracker»1. НАЗНАЧЕНИЕ И ЦЕЛИ СИСТЕМЫСистема «RiskTracker» предназначена для автоматизации процессов идентификации, качественной и количественной оценки рисков, ведения сквозного аудита изменений, контроля выполнения защитных мероприятий (контрмер) и формирования консолидированной отчетности.Цели внедрения:Исключение ручного учета рисков в разрозненных таблицах.Обеспечение 100% прозрачности изменений параметров рисков (кто, когда и что изменил).Контроль исполнительской дисциплины сотрудников при реализации контрмер.Предоставление руководству оперативной аналитики в разрезе бизнес-процессов, проектов и ИТ-систем.2. СТЕК ТЕХНОЛОГИЙ И АРХИТЕКТУРНЫЕ ПРИНЦИПЫПроект реализуется в рамках монорепозитория со строгим разделением на бэкенд и фронтенд.  Бэкенд:Язык программирования: Golang (Go).Архитектура бэкенда: Clean Architecture (разделение на слои: delivery/http $\rightarrow$ usecase $\rightarrow$ repository).  Базовые библиотеки: chi или gin для маршрутизации, pgx или sqlx для работы с БД.Фронтенд:Фреймворк: Next.js (React) с использованием App Router.  Стилизация и UI: TailwindCSS + Shadcn UI (на базе Radix Primitives).Управление состоянием и запросы: TanStack Query (React Query) + Axios / Fetch API.  База данных: PostgreSQL (версии 15 и выше).Интерфейсный контракт: Swagger / OpenAPI 3.0. Спецификация хранится в файле api/swagger/swagger.yaml на бэкенде и используется для автогенерации TypeScript-типов на фронтенде.  3. СТРУКТУРА ПРОЕКТА (КАТАЛОГИ СИСТЕМЫ)Plaintextrisktracker/

ПОЛНАЯ РЕЛЯЦИОННАЯ СТРУКТУРА ДАННЫХ (POSTGRESQL)В системе определены следующие перечисления (ENUM):risk_level: 'LOW', 'MEDIUM', 'HIGH', 'CRITICAL'risk_status: 'IN_PROGRESS', 'COMPLETED'user_role: 'USER', 'ADMIN'countermeasure_target: 'CAUSE', 'CONSEQUENCE'risk_object_type: 'IT_SYSTEM', 'PROJECT', 'PROCESS'Схема таблиц со всеми ограничениями (Constraints) и индексами:SQLCREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- 1. Пользователи
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    full_name VARCHAR(255) NOT NULL,
    role user_role NOT NULL DEFAULT 'USER',
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- 2. Справочник: Объекты риска
CREATE TABLE risk_objects (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name VARCHAR(255) UNIQUE NOT NULL,
    object_type risk_object_type NOT NULL,
    description TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- 3. Риски
CREATE TABLE risks (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    status risk_status NOT NULL DEFAULT 'IN_PROGRESS',
    title VARCHAR(255) NOT NULL,
    target_id UUID NOT NULL REFERENCES risk_objects(id) ON DELETE RESTRICT,
    owner_id UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT,

    -- Оценка (Секция 3)
    probability risk_level NOT NULL,
    impact risk_level NOT NULL,
    financial_loss TEXT,
    reputational_loss risk_level,
    legal_consequences INT CHECK (legal_consequences BETWEEN 1 AND 5),
    comment TEXT,

    -- Автовычисляемые поля Секции 2
    max_cause_probability risk_level,
    max_consequence_probability risk_level,

    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- 4. Причины риска
CREATE TABLE risk_causes (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    risk_id UUID NOT NULL REFERENCES risks(id) ON DELETE CASCADE,
    name VARCHAR(255) NOT NULL,
    description TEXT,
    probability risk_level NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- 5. Последствия риска
CREATE TABLE risk_consequences (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    risk_id UUID NOT NULL REFERENCES risks(id) ON DELETE CASCADE,
    name VARCHAR(255) NOT NULL,
    description TEXT,
    probability risk_level NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- 6. Контрмеры
CREATE TABLE countermeasures (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    risk_id UUID NOT NULL REFERENCES risks(id) ON DELETE CASCADE,
    target_type countermeasure_target NOT NULL,
    cause_id UUID REFERENCES risk_causes(id) ON DELETE CASCADE,
    consequence_id UUID REFERENCES risk_consequences(id) ON DELETE CASCADE,
    description TEXT NOT NULL,
    assignee_id UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    deadline TIMESTAMP WITH TIME ZONE NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT check_deadline CHECK (deadline > created_at),
    CONSTRAINT check_target_integrity CHECK (
        (target_type = 'CAUSE' AND cause_id IS NOT NULL AND consequence_id IS NULL) OR
        (target_type = 'CONSEQUENCE' AND consequence_id IS NOT NULL AND cause_id IS NULL)
    )
);

-- 7. Журнал изменений сущностей (Entity-level Audit Trail)
CREATE TABLE entity_audit_logs (
    id BIGSERIAL PRIMARY KEY,
    entity_type VARCHAR(50) NOT NULL,      -- 'RISK', 'COUNTERMEASURE', 'RISK_OBJECT'
    entity_id UUID NOT NULL,               -- ID корневой сущности
    action_type VARCHAR(20) NOT NULL,      -- 'CREATE', 'UPDATE', 'DELETE'
    changed_by_user_id UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    timestamp TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    old_state JSONB,                       -- NULL при CREATE
    new_state JSONB                        -- NULL при DELETE
);

-- Индексы для оптимизации выборок и аналитики
CREATE INDEX idx_risks_search ON risks(target_id, status);
CREATE INDEX idx_risks_owner ON risks(owner_id);
CREATE INDEX idx_countermeasures_assignee ON countermeasures(assignee_id, deadline);
CREATE INDEX idx_audit_history ON entity_audit_logs(entity_type, entity_id, timestamp DESC);
5. ПОЛНАЯ СПЕЦИФИКАЦИЯ ЭНДПОИНТОВ (REST API ROUTES)Все запросы, кроме авторизации, требуют передачи заголовка Authorization: Bearer <JWT_TOKEN>.5.1. Аутентификация и пользователиPOST /api/v1/auth/login — Аутентификация. Принимает {email, password}. Возвращает JWT-токен и устанавливает сессионную куку.POST /api/v1/auth/logout — Инвалидация сессии.GET /api/v1/users/me — Получение профиля текущего пользователя (ID, ФИО, Email, роль).GET /api/v1/users — Получение списка всех пользователей (для выпадающих списков ответственных).POST /api/v1/admin/users — [Admin] Создание пользователя {email, full_name, password, role}.PUT /api/v1/admin/users/:id/password — [Admin] Принудительная смена пароля.5.2. Справочник объектов рискаGET /api/v1/objects — Получение списка всех объектов.POST /api/v1/admin/objects — [Admin] Создание нового объекта {name, object_type, description}.PUT /api/v1/admin/objects/:id — [Admin] Редактирование объекта.5.3. Риски (Секции 1, 2, 3)GET /api/v1/risks — Получение списка рисков с пагинацией. Обычный пользователь получает только риски, где он owner или assignee в контрмерах.GET /api/v1/risks/:id — Получение полной древовидной структуры риска (включая массивы причин, последствий и привязанных контрмер).POST /api/v1/risks — Создание риска. Тело запроса включает базовую шапку и оценку.PUT /api/v1/risks/:id — [Slide-over API] Полное обновление рисков. Принимает обновленный снапшот всех секций.PATCH /api/v1/risks/:id/status — Изменение статуса (IN_PROGRESS $\leftrightarrow$ COMPLETED).5.4. Элементы Секции 2 и КонтрмерыPOST /api/v1/risks/:id/causes — Добавление причины.POST /api/v1/risks/:id/consequences — Добавление последствия.POST /api/v1/countermeasures — Создание контрмеры с жесткой привязкой к cause_id или consequence_id.PUT /api/v1/countermeasures/:id — Редактирование контрмеры (изменение текста, дедлайна или ответственного).DELETE /api/v1/countermeasures/:id — Удаление контрмеры.5.5. Аналитика, Отчеты и ЛогиGET /api/v1/reports/summary — Получение данных для конструктора отчетов.Query параметры: target_id, assignee_id, status, from_date, to_date.GET /api/v1/audit/logs/:entity_type/:entity_id — [Admin] Получение таймлайна изменений конкретной сущности.POST /api/v1/audit/rollback — [Admin] Инициирование ручного отката данных на основе выбранного old_state.6. ПОДРОБНЫЕ СЦЕНАРИИ ИСПОЛЬЗОВАНИЯ (USE CASES / ПОЛЬЗОВАТЕЛЬСКИЕ ИСТОРИИ)Сценарий 1: Создание и оценка риска (Пользователь)Пользователь переходит в раздел «Риски» и нажимает кнопку «Создать риск».Заполняет Секцию 1: вводит название, выбирает из справочника «Объект риска» (система не дает ввести текст руками) и назначает владельца.Заполняет Секцию 2: добавляет две причины возникновения и одно последствие. Для каждого элемента выставляет уровень вероятности (например, HIGH).Заполняет Секцию 3: выставляет общую вероятность, влияние, вносит финансовые и законодательные оценки. Нажимает «Сохранить».Система: Бэкенд сохраняет записи, триггер автоматически вычисляет наивысшие значения (max_cause_probability = HIGH), а модуль аудита делает запись в entity_audit_logs с action_type = 'CREATE' и old_state = NULL.Сценарий 2: Быстрое изменение риска из отчета через Slide-over DrawerПользователь (или Администратор) находится на странице «Конструктор отчетов» и анализирует таблицу рисков.В строке критического риска пользователь нажимает на иконку ✏️.С правой стороны экрана плавно выезжает панель Slide-over Drawer (Next.js выполняет асинхронный запрос GET /api/v1/risks/:id и мгновенно рендерит форму). Левая часть экрана с отчетом затеняется.Пользователь прямо в Drawer меняет статус риска на COMPLETED и дополняет финансовую оценку. Нажимает «Сохранить изменения».Система: Отправляется запрос PUT /api/v1/risks/:id. Бэкенд фиксирует новое состояние, создает лог изменений. Фронтенд закрывает Drawer, а библиотека TanStack Query инвалидирует кэш страницы отчета — строка в таблице обновляется на лету, сохраняя все примененные до этого фильтры.Сценарий 3: Контроль контрмер и дедлайнов (Генерация отчета)Администратор заходит в раздел «Отчеты» и выбирает в фильтре «Ответственное лицо» конкретного сотрудника.Интерфейс отображает аналитический блок:Количество просроченных контрмер (Expired).Таймлайн сгорающих сроков.Ниже рендерится список рисков. Администратор видит контрмеру, срок которой просрочен (deadline < текущее_время при статусе риска IN_PROGRESS).Администратор может прямо отсюда открыть Drawer риска и переназначить контрмеру на другого сотрудника.Сценарий 4: Аудит изменений и ручной откат состояния (Администратор)Администратор открывает карточку риска и переходит во вкладку «Аудит изменений».Интерфейс отображает список версий, сгенерированных на основе entity_audit_logs. Каждая запись интерактивна: при клике подсвечивается разница (Diff) между old_state и new_state.Администратор видит ошибочное изменение, сделанное пользователем день назад.Администратор нажимает «Восстановить до этой версии». Система берет данные из сохраненного old_state выбранного лога и отправляет команду на бэкенд для перезаписи текущих полей риска.7. ВИЗУАЛЬНОЕ ОФОРМЛЕНИЕ И ИНТЕРФЕЙСНЫЕ РЕШЕНИЯ (UI/UX)Интерфейс адаптирован под бизнес-аналитиков и риск-менеджеров. Цветовая гамма строгая: #1E293B (серый-графит) для навигации, #FFFFFF для рабочих областей, #F3F4F6 для фона страниц.Матрица рисков 4х4: На главном экране Dashboard размещается тепловая карта рисков, где оси — это Вероятность наступления и Влияние риска (Секция 3). Клетки окрашены градиентом от зеленого до ярко-красного. Числа внутри клеток показывают количество рисков данной категории. Клик на клетку фильтрует главную таблицу.Двухколоночный редактор Секции 2: При создании/редактировании риска Причины и Последствия располагаются параллельно (в две колонки). Это позволяет пользователю визуально сопоставить, какая причина какое последствие генерирует.Компонент Slide-over Drawer: Реализуется с помощью компонента Sheet из библиотеки Shadcn UI (под капотом Radix UI Dialog). При вызове он блокирует скролл основной страницы, а плавность анимации обеспечивается средствами CSS/Tailwind (translate-x-0).8. НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯБезопасность и хранение сессий: Передача токенов осуществляется исключительно через заголовки, а сам токен авторизации (Refresh token) сохраняется в куках с флагами Secure, HttpOnly и SameSite=Strict для предотвращения кражи сессии через XSS/CSRF атаки.Защита данных от удаления: В системе полностью отсутствует физическое удаление рисков (оператор DELETE для таблицы risks на бэкенде не вызывается). Завершение работы с риском — это перевод в статус COMPLETED.Производительность: Скорость обработки запросов на генерацию отчетов бэкендом не должна превышать 200 мс для баз данных объемом до 100 000 рисков за счет использования составных индексов.
